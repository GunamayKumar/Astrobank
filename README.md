## 1. What this project is

Astrobank is an evidence-based astrology research platform. It is explicitly
**not** a conventional astrology app — its core design principle is that it
keeps six categories of data strictly separated and never lets them blend:

1. **Immutable factual data** — e.g. birth data, chart facts. Once created, never edited.
2. **Calculated astrological data** — derived from facts by an astrology engine (planetary positions, houses, aspects, yogas, doshas, etc.). Stored separately from facts.
3. **Human observations** — interpretations written by users. Never overwritten by AI.
4. **AI observations** — interpretations generated by AI models. Additive only, never overwrite human observations.
5. **Research analytics** — datasets, consensus, prediction outcome tracking. Derived and read-only.
6. **Knowledge articles** — editorial content, backed by evidence/references.

The long-term goal is to support statistical research into astrology (does it
actually predict anything?) while also functioning as a community platform
where people can log charts, add observations, and discuss them.

## 2. Non-negotiable rules (frozen — do not redesign)

These come from months of upstream analysis/design and are **frozen** unless a
human explicitly says otherwise in a new instruction. Do not "improve" or
reinterpret them.

**Architecture**
- Clean Architecture, feature-based organization, high cohesion / low coupling.
- Business logic lives in the **Application** layer. Controllers are thin —
  orchestration only, never business rules.
- Persistence (EF Core) is isolated behind Application-layer abstractions.
  EF Core types must never leak outside `Astrobank.Persistence` (except the
  Web composition root wiring it up via DI).
- The astrology engine is fully abstract behind `IAstrologyEngine`. The
  Application layer depends on that interface, never on a concrete engine
  (Parashara, KP, Jaimini, Nadi, Lal Kitab — these are future pluggable
  implementations living in `Astrobank.AstrologyEngine`).
- Nullable reference types enabled everywhere. Prefer async. Constructor DI only.

**Business rules**
- A chart has exactly one owner. Only the owner may edit it.
- Administrators cannot modify chart facts. Facts are immutable, full stop —
  not even for admins.
- Calculated astrological values are stored separately from facts, and facts
  and interpretations (observations) must never be mixed in the same table
  or the same mental model.
- AI observations never overwrite human observations — they are always
  additive, separate records.
- Community comments never modify chart data.
- Research data (datasets, results, consensus) is read-only from the
  application's point of view — it's derived/computed, not hand-edited.
- Prediction outcomes are stored separately from predictions themselves.
- Consensus is derived data, computed from observations/outcomes — not
  entered directly.
- Knowledge articles must be supported by evidence/references, not
  freestanding opinion.

**Database**
- The full schema (Users, UserProfiles, Roles, Permissions, Charts,
  ChartImages/Files/Events/Tags/Access/AuditLog, ChartObservations,
  ResearchDatasets/Jobs/Results, PlanetaryPositions/HousePositions/Aspects/
  Strengths, Yogas, Doshas, KnowledgeArticles, AIModels/AIAnalysisHistory/
  AIRecommendations, FeatureFlags, ApplicationSettings, and supporting lookup
  tables) is already designed and **frozen**. Do not redesign it. If a table
  or column appears to be missing something needed for a task, ask — don't
  invent it.

**Workflow rules for agents**
- Work **incrementally**, one module at a time, as explicitly assigned by the
  human. Do not build unrelated modules "while you're at it."
- Each unit of work = one logical commit: commit message, files added, files
  modified, folder structure, complete code, build notes. No half-finished
  scaffolding left uncommitted.
- No placeholder/demo code unless specifically asked for. Code must compile.
- Never invent business rules. Never redesign the architecture or schema.
  If a requirement is ambiguous, ask a clarifying question rather than
  guessing and moving on.
- XML documentation, proper logging (Serilog), validation (FluentValidation),
  and DI registration are expected as each relevant module is implemented —
  not skipped "for now" once a module actually needs them.

## 4. Solution structure

```
Astrobank.sln
Directory.Build.props      # shared MSBuild settings (net8.0, nullable, etc.)
global.json                 # pins .NET SDK to 8.0.100, rollForward latestMinor
.gitignore

src/
  Astrobank.Domain/          # entities, value objects, domain enums/events. No framework deps.
  Astrobank.Shared/          # cross-cutting primitives (Result types, constants, extensions). No deps on anything else.
  Astrobank.Application/     # use cases, DTOs, interfaces/abstractions (incl. IAstrologyEngine contract), FluentValidation validators, AutoMapper profiles.
  Astrobank.Persistence/     # EF Core DbContext, entity configurations, migrations, repositories. Implements Application abstractions.
  Astrobank.Infrastructure/  # non-persistence external integrations: storage, email, Serilog sinks, third-party clients.
  Astrobank.AstrologyEngine/ # IAstrologyEngine implementations (Parashara, KP, Jaimini, Nadi, Lal Kitab — none implemented yet).
  Astrobank.Research/        # research datasets, consensus calculation, prediction validation, ML dataset export.
  Astrobank.AI/               # AI observation generation, AI recommendations, model benchmarking.
  Astrobank.Web/               # ASP.NET Core MVC composition root. Only project allowed to reference Persistence + Infrastructure directly.

tests/    # empty — no test projects yet
database/ # empty — for SQL scripts / DACPAC / migration artifacts, not yet populated
docs/     # empty — for design docs
```

### Project reference graph (already wired in the `.sln` / `.csproj` files)

```
Shared  ←  Domain  ←  Application  ←  Persistence
                            ↑    ↖     Infrastructure
                            │      ↖   AstrologyEngine
                            │        ↖ Research
                            │          ↖ AI
                            └──────────── Web (references everything)
```

- `Domain` and `Shared` have zero outward project dependencies.
- `Application` depends only on `Domain` + `Shared`. **This is where
  `IAstrologyEngine` and repository/unit-of-work interfaces are meant to be
  declared** once that work starts — not in `AstrologyEngine` or
  `Persistence` themselves.
- `Persistence`, `Infrastructure`, `AstrologyEngine`, `Research`, `AI` each
  depend on `Application` + `Domain` + `Shared`, and never on each other.
- `Web` references every project and is the only place concrete
  implementations get registered against Application abstractions via DI
  (the composition root).

### Current state

What exists right now:
- `Astrobank.sln` with all 9 projects, correctly cross-referenced as above.
- Initial domain entities for the Users module (User, UserProfile, Role, Permission, Country) implemented in `Astrobank.Domain`.
- `AstrobankDbContext`, Fluent API entity configurations, and initial EF Core migration for the Users module created in `Astrobank.Persistence`. Registered in `Astrobank.Web`.
- `Astrobank.Web` has the minimum needed for an ASP.NET Core MVC app to
  exist and build:
  - `Program.cs` — `AddControllersWithViews()` + default route only. No
    Identity, no EF Core `DbContext` registration, no Serilog, no
    FluentValidation/AutoMapper, no `IAstrologyEngine` registration. These
    are explicitly deferred to their own future commits.
  - `appsettings.json` / `appsettings.Development.json` — bare defaults, no
    connection strings yet (SQL Server connection string will be added when
    `Persistence`/`DbContext` work begins).
  - `Properties/launchSettings.json` — standard Kestrel + IIS Express
    profiles.
  - Empty placeholder folders (tracked via `.gitkeep`): `Controllers/`,
    `Views/Home/`, `Views/Shared/`, `wwwroot/css`, `wwwroot/js`, `wwwroot/lib`.
- `tests/`, `database/`, `docs/` exist as empty top-level folders (per the
  agreed solution structure) but contain no projects/content yet.
- `Directory.Build.props` centralizes `TargetFramework=net8.0`,
  `Nullable=enable`, `ImplicitUsings=enable`, `GenerateDocumentationFile=true`
  so individual `.csproj` files stay minimal.

**Known caveat:** the sandbox this was built in did not have network access
to a .NET package mirror, so the `.sln`/`.csproj` files were hand-authored
rather than generated via `dotnet new`/`dotnet sln add`, and **have not yet
been verified with an actual `dotnet build`**. First thing to do in this
repo is run:

```bash
dotnet restore
dotnet build
```

and fix anything that doesn't compile before adding more code on top.

## 6. How to work in this repository (for agents and humans)

1. **Work on one module at a time**, as assigned. Don't jump ahead into
   unrelated modules even if it seems efficient.
2. **Never touch the frozen items** in Section 2 (architecture shape,
   business rules, database schema) without an explicit instruction to do so.
3. **Respect the reference graph** in Section 4. If a change would require
   adding a `ProjectReference` that isn't in that graph (e.g. `Domain` →
   `Persistence`), that's a signal something is architecturally wrong —
   stop and flag it rather than adding the reference.
4. **Keep facts/calculated data/observations/research/knowledge separated**
   at the entity and table level, per Section 2. When in doubt about which
   category something belongs to, ask.
5. **Match the existing commit format**: commit message, files added, files
   modified, folder structure, complete code, build notes. Don't leave
   partial/uncommitted work.
6. **Production-quality only**: XML docs, logging, validation, DI — no demo
   code, no `TODO` stubs left in place of real implementation, unless a
   placeholder was explicitly requested.
7. **If a requirement is unclear or missing from this README or the frozen
   design**, ask a clarifying question instead of inventing a business rule
   or guessing at schema details.
